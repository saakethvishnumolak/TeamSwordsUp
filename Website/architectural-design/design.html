Arch Design

Principles - Description/Components

Profile Differentiation

  • It should be possible to configure what type of profile you manage, and safely/accurately set up both types
  • Components: Profile role selection, Sport/Sport type selection, Collegiate account verification, Profile creation and display UI

Data Management

  • It should be possible to upload and delete user data in the app
  • Components: Profile-realted data storage, Statistic storage, Data timestamping, Statistic data date tracking, Video data storage, Video data sharing, 3rd party import and export; Data management UI, Video streaming UI

Comparisons and Calculations

  • It should be possible to compare the data of different profiles against one another, and receive the results in an easy to read fashion
  • Components: Comparison type selection - player to player or team to team, Statistic data retrieval, Internal calculations, Comparison results UI, Collegiate account verification

Security and Privacy

  • It should be possible to prevent user data from being lost or stolen within the software to the best of the program's ability
  • Components: Collegiate personal data defenses, Video-related data defenses, Data backups via cloud based storage systems, Account login and password system

Architectural Section - Components

App Interface

  • Profile creation and display UI, Data management UI, Video streaming UI, Comparison selection and results UI

Internal Data

  • Profile-related data storage, Statistic storage

Access and Management

  • Comparison type selection - player to self ONLY, Internal calculations, Statistic timestamping, Statistic date tracking, 3rd party import and export

Cloud-based Data Access

  • Account login and password system, Comparison type selection - team to team and player to player, Separate user data retrieval, Video data storage, Video data sharing, Internal data backups

Secured Server Data

  • Collegiate personal data defenses, Video-related data defenses, Collegiate account verification
Distribution Architecture
Narrative
  • What database will we use?
    • Public Cloud based system, with some of the app’s data being stored within the user’s device. The cloud system is planned to be a multi-tenant system, in order to keep database costs low.
    • While most user data will be kept in a single database, secure information (such as login info, and emails used for authenticating collegiate profiles) will be kept in one of 4-5 separate databases, depending on the user’s timezone when they first create an account.(US Time Zones only)
  • What delivery platform will we develop for?
    • Microsoft Azure - good for setting up businesses with a large number of users, granting the ability to support all of our clients across the US
  • What server platform will we use?
    • Linux - offers more flexibility than platforms like Windows, through its wider array of languages and open source software.
  • What open-source products will be part of the system?
    • API, Ubuntu
  • What development tools will you use?
    • React/nodejs, Docker, Virtual Machine
